  1. User Story 1: As a user, I want Athena to assist me in generating schedules so that the time taken to create a schedule is cut down.
  1. In our problem of study, we stated that scheduling in Athena is confusing and inefficient because it takes a long time to create one’s schedule. With the current set-up on Athena, students have to do the work of figuring out what sections don’t overlap on their own. From this problem, we generated this specific user story with the aim of cutting down on the labor for scheduling planning. Our prototype addresses this by providing a way for the user to automatically generate schedules, thereby removing the need for the student to do the thinking themselves. Specifically, it includes the design for a system where the user can enter in their classes, block off unavailable times, and get schedule options generated for them. On the class search tab, students can search for and select classes which are then added into the selected classes list. On the preferences tab, students can mark off blocks of time that they want to remain open by clicking with the mouse. These blocks are highlighted in red when selected and white when deselected. After pressing the “generate schedules” button, the user can then view the different potential schedules that have been generated for them and select the one that they want to register for on the schedules tab. The three tabs are at the top of the screen so that the user can switch between the three at their leisure, allowing them to make adjustments to their selected classes and time preferences as they go. If there’s a combination of schedules and times that has no schedule possibilities, then the user is given a notification alerting them that they need to make some changes. This design will significantly cut down on time taken to create a schedule because the user no longer has to pick the correct classes themselves and make sure that class times work out with each other.

  1. User Story 2: As a user, I want the option to select available classes as I type in a class name so that I can easily multi-select classes through their names.

  1. Another issue we found in our problem of study is that Athena’s class sign up process is difficult to navigate around because nothing appears until you press enter, and the process of adding classes to a schedule can be tedious with no way to add or remove multiple classes at once. Attempting to compare multiple class times to plan your schedules is also tedious, because there is no way to switch back and forth between different sections of a class without removing the class and adding it again. This is why we came up with a user story with the aim of having an easier class sign up process, specifically through multi-select. Our design combined two ideas: having a dropdown menu that lets the user select multiple classes at once, and having a side menu where the user can view the selected classes and change which section of the class they want to register for.

Athena sign up design:

In our prototype, we redesigned the class sign up process by using a dropdown with checkboxes that appears as the user types class names in order to fulfill  the user story’s need to multi-select classes as they are typing in class names. This way, the user can select their desired course prefix, then select multiple courses at once.

The user is able to see the classes get added to the selected class list as they select them.

To allow for further customization, our prototype has a dropdown with radio buttons for selecting different class times and seeing how they look on the calendar. With this design, the user can select both different classes and different available class times.

If the user decides that they want to remove the class, they can uncheck the box and it will be removed from their schedule.

  1. As a user, I want the windows on the registration page to display the most relevant information more prominently so that I can register for my classes easier.

  1. In our problem of study, we also found that important information such as prerequisites and class capacity were unable to be easily found. Therefore, we came up with the user story that we wanted more prominent ways to display relevant information. In order to fulfill this, we designed pop ups that appear when the user hovers with their mouse or attempts to add a particular class to their schedule. This ensures that urgent and important information is easily viewable while the student is still planning their schedule, so they can make informed decisions before they go to register. For our prototype, we designed two kinds of examples, one if a prerequisite is not fulfilled, and one if a CRN is full. Both function the same way, but they display different messages to give an idea of what different alert messages might look like. We also implemented a dropdown menu similar to the one we made for the second user story prototype. With this dropdown menu, the user can see which classes they should avoid adding to their schedule by noting which ones are highlighted in red and marked with an X. The hover feature forwarns the user before they attempt to add it to their schedule, and a banner shows up in the center of the screen if they try to add it anyway.

Part B:

Our research question is “Do students prefer to register for classes on our revamped versions of Athena better than the current version?” Based on this research question, we decided to use a blend of an experiment and a survey for our methodology. Because our research question is based on user experience rather than a tangible outcome, an experiment alone would not be sufficient in addressing our question. Also, if we just sent out a survey with video demonstrations instead of having the students interact with the website or prototype, the results may not be accurate to how the students would feel if they had interactive capabilities. According to an article on user-experience research methods by Christian Rohrer, when assessing a product, or measuring “product performance against itself or its competition,” surveys and A/B testing are the best approaches to the research ( 1 ). A/B testing is a procedure for testing designs based on randomly assigned groups of users each interacting with different designs and “measuring the effect of these assignments on user behavior” ( 1 ).   For these reasons, we will be conducting an experiment where the goal of the subject is to register for a certain schedule of classes, and a survey will be administered afterwards with questions regarding how the user felt about their experience. We will also utilize the A/B testing style by having each participant test one prototype.

For our testing procedure, the first step is to find participants for the study. We are limiting the scope of participants to students at the University of Georgia. This is because, according to A Human Activity Approach to User Interfaces , it is important for the user interface to not be seen independently of the use  activity ( 2 ). This means that a user interface is best used by those who will actually be using it in real life, so students who have used Athena would be best suited for the study. We could send out a short survey to gather interest on the UGA listserv, similar to other research studies at the university. Once we have a sufficient number of people interested in the study, we could set up a Zoom call to conduct the study while adhering to pandemic safety regulations. Each of the subjects will be randomly assigned one of four groups: the control group, prototype 1 (P1), prototype 2 (P2), and prototype 3 (P3). The three prototype groups will conduct the experiment using the prototypes from the first, second, and third user stories respectively. The control group will use a prototype of the Athena registration page as it is. Before the experiment, a consent form will be sent to each of the participants while on the Zoom call. The form will detail how the data will be anonymous, how no personal information will be collected from the participant, how the study is completely voluntary and the participant can stop at any time, and any other concerns that the participant may have. The test proctor will be there to answer any questions the participant may have about the consent form.

Once the consent form has been given, the proctor will then send a link to a webpage to the subject based on the group that the participant has been assigned. On the home page, there will be a video demonstration on how to register for a class in the specific prototype.  A video demonstration is important to include according to a study done by C. Plaisant and B. Schneiderman, which found that “recorded demonstrations were quite effective, allowing users to get started quickly and complete the tasks. Most users seemed to have learned all aspects of the interface within a few minutes and were able to focus on the task immediately”.  ( 4 ) We will follow the specific guidelines outlined in this paper in order to produce a video that will be understandable and useful to the user, such as providing procedural instructions, keeping segments brief, ensuring tasks are simple, and using voice narration while coordinating the demonstrations with textual documentation. ( 4 )  The next page will be a description of the task the participant is required to do. This task will be registering for a given schedule of three classes. The only requirement for this task is that the schedule must be valid (i.e. the classes must not overlap, the course sections must not be full, and the prerequisites for the class must be completed). Throughout this process and before the test begins, the participants will have the opportunity to ask the proctor any questions they have about the information they received. Once the participant is ready to start the experiment, the user clicks the start button and the beginning web page for each prototype is given to the user.

Once the user is done registering for their classes successfully, they can hit a “Finish” button and the test will be concluded. The proctor will then send a follow-up survey to the participant for them to complete. The questions for this survey are outlined below. All of the survey responses are quantitative, and they will all be analyzed as such. Each participant's responses will be anonymous. According to Christian Rohrer, the best way to approach responses that assess a product in relation to competition is “mainly quantitative” research  ( 1 ). This is why the survey questions are predominantly quantitative. The data will be sent to a Google Sheets page automatically so that the data can be viewed easily and can be compared and analyzed easily. Since all of our questions require the user to respond on a scale, either numerical or on a scale of satisfaction, we would analyze the data using mode, mean and bar charts. ( 5 ) As shown in the questions below, the analysis of the survey will help answer our research question by showing the average user experience results for each of the prototypes in relation to the original Athena. If one or more of the prototypes are preferred to Athena’s current registration page, then the answer to our question would be yes.

Survey Questions

Overall, how did you feel about using this type of class registration?

(Strongly Dissatisfied, Dissatisfied, Slightly Dissatisfied, Neutral, Slightly Satisfied, Satisfied, Strongly Satisfied)

On a scale of 1-10, how would you rate the ease of moving between different screens?

On a scale of 1-10, how would you rate the clarity of prompts while using the website?

On a scale of 1-10, how would you rate the availability of information presented?

How would you rate the time it took to generate a schedule?

(Faster than Average, Average, Slower than Average)

How helpful did you find the provided demo video?

(Very Unhelpful, Unhelpful, Slightly Unhelpful, Neutral, Slightly Helpful, Helpful, Very Helpful)

If you had any questions for the proctor, how helpful were they in answering your questions?

(No Questions, Very Unhelpful, Unhelpful, Slightly Unhelpful, Neutral, Slightly Helpful, Helpful, Very Helpful)

How would you feel about using this software in the future?

(Strongly Dissatisfied, Dissatisfied, Slightly Dissatisfied, Neutral, Slightly Satisfied, Satisfied, Strongly Satisfied)
